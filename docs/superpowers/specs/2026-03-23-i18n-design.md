# Internationalisation (i18n) — Design Spec

**Date:** 2026-03-23
**Status:** Approved

---

## Goal

Add Polish translation, auto-detect the browser language on first use, and let users override the language in Settings.

---

## Supported Languages

| Code | Language |
|------|----------|
| `en` | English  |
| `pl` | Polish   |

Auto-detect resolves to `pl` when `navigator.language.startsWith('pl')`, otherwise `en`.

---

## Architecture

### Locale files

| File | Role |
|------|------|
| `src/lib/i18n/types.ts` | Exports `Translations` type (inferred from `en.ts`) |
| `src/lib/i18n/en.ts` | English strings — exports `const en: Translations` |
| `src/lib/i18n/pl.ts` | Polish strings — exports `const pl: Translations` |
| `src/lib/i18n/index.ts` | Exports `t` as `Readable<Translations>`, derived from `settingsStore` |

`types.ts` extracts the type from `en.ts` so both locale files are statically checked to have the same shape. If a key is missing from `pl.ts`, TypeScript will error at compile time.

### Locale resolver (`index.ts`)

```ts
import { derived } from 'svelte/store';
import { settingsStore } from '../stores/settingsStore';
import { en } from './en';
import { pl } from './pl';

function resolve(lang: 'auto' | 'en' | 'pl'): Translations {
  const effective = lang === 'auto'
    ? (navigator.language.startsWith('pl') ? 'pl' : 'en')
    : lang;
  return effective === 'pl' ? pl : en;
}

export const t = derived(settingsStore, s => resolve(s.language));
```

### Settings type

Add `language: 'auto' | 'en' | 'pl'` to the `Settings` interface in `src/lib/types.ts`.
Default: `'auto'`.

### SettingsSheet UI

New segmented-control row (same pattern as `appearance`):

- Label: language setting name (localised)
- Segments: `Auto`, `EN`, `PL`
- Placement: between the Wind Speed Unit row and the Appearance row

### i18n strings added to both locales

```
language        — section label
languageNames   — { auto: 'Auto', en: 'English' / 'Angielski', pl: 'Polski' / 'Polish' }
```

## Component changes

All components that currently import `t` from `'../i18n/en'` change to:

```ts
import { t } from '../i18n';
```

All usages of `t.foo` in templates and script blocks change to `$t.foo`.

**Affected components:**
- `AppHeader.svelte`
- `SummaryStrip.svelte`
- `HeatmapCanvas.svelte`
- `TimeSlider.svelte`
- `ThresholdFooter.svelte`
- `SettingsSheet.svelte`
- `ErrorBanner.svelte`

---

## Data flow

```
settingsStore.language
       ↓
  resolve(lang) → Translations object
       ↓
  t (Readable<Translations>)
       ↓
  $t.foo  ← all components
```

---

## Error handling

- If `navigator.language` is unavailable (SSR / unusual env), `resolve('auto')` falls back to `en`.
- No async loading — both locale files are bundled at build time.

---

## Out of scope

- Pluralisation rules
- RTL support
- Number/date formatting per locale
- More than two languages (extensible but not planned)

---

## Testing

No new unit tests required. The `resolve` function is a pure function and trivially correct. Existing 47 tests must continue to pass after the change.
