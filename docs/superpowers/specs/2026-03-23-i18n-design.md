# Internationalisation (i18n) — Design Spec

**Date:** 2026-03-23
**Status:** Approved

---

## Goal

Add Polish translation, auto-detect the browser language on first use, and let users override the language in Settings.

---

## Supported Languages

| Code | Language |
|------|----------|
| `en` | English  |
| `pl` | Polish   |

Auto-detect resolves to `pl` when `navigator.language.startsWith('pl')`, otherwise `en`. Any unrecognised persisted value (e.g. a future code from an old build) silently falls back to `en`.

---

## Architecture

### 1. `src/lib/types.ts` — new type + updated interface

Add `AppLanguage` alongside the existing `WindUnit` and `AppAppearance`:

```ts
export type AppLanguage = 'auto' | 'en' | 'pl';
```

Add `language: AppLanguage` to `Settings`:

```ts
export interface Settings {
  thresholdKmh: number;
  unit: WindUnit;
  appearance: AppAppearance;
  refetchRadiusKm: number;
  language: AppLanguage;   // NEW
}
```

### 2. `src/lib/stores/settingsStore.ts` — update `defaults()`

```ts
function defaults(): Settings {
  return { thresholdKmh: 25, unit: 'kmh', appearance: 'auto', refetchRadiusKm: 5, language: 'auto' };
}
```

The existing `load()` spreads `defaults()` over the stored value, so users with an older persisted object (missing `language`) correctly receive `'auto'`.

### 3. `src/lib/i18n/en.ts` — rename export, add new keys, add import

Change `export const t = { … }` to `export const en = { … }`.

Update the import line to include `AppLanguage`:
```ts
import type { WindUnit, AppAppearance, AppLanguage } from '../types';
```

Add two new keys to the object:
```ts
language: 'Language',
languageNames: { auto: 'Auto', en: 'English', pl: 'Polish' } as Record<AppLanguage, string>,
```

### 5. `src/lib/i18n/types.ts` — new file

> **Order dependency:** Create this file only after step 3 (`en.ts`) has been updated with the two new keys. `Translations` is derived as `typeof en`, so it must include `language` and `languageNames` before `pl.ts` (step 4) is type-checked against it.

```ts
import { en } from './en';
export type Translations = typeof en;
```

`AppLanguage` lives in `src/lib/types.ts` (not here) to avoid circular imports. The import chain is: `en.ts → ../types`, `pl.ts → ../types`, `types.ts → en.ts` — no cycle.

### 6. `src/lib/i18n/index.ts` — new file

```ts
import { derived } from 'svelte/store';
import { settingsStore } from '../stores/settingsStore';
import { en } from './en';
import { pl } from './pl';
import type { Translations } from './types';
import type { AppLanguage } from '../types';

function resolve(lang: AppLanguage): Translations {
  const effective =
    lang === 'auto'
      ? (typeof navigator !== 'undefined' && navigator.language.startsWith('pl') ? 'pl' : 'en')
      : lang;
  return effective === 'pl' ? pl : en;
}

export const t = derived(settingsStore, s => resolve(s.language));
```

## Component migration

Run `grep -r "i18n/en" src/` to find all affected files. As of this writing, five files need updating:

| File | Current import | New import |
|------|---------------|------------|
| `src/App.svelte` | `'./lib/i18n/en'` | `'./lib/i18n'` |
| `src/lib/components/AppHeader.svelte` | `'../i18n/en'` | `'../i18n'` |
| `src/lib/components/SummaryStrip.svelte` | `'../i18n/en'` | `'../i18n'` |
| `src/lib/components/DesktopUtilityRail.svelte` | `'../i18n/en'` | `'../i18n'` |
| `src/lib/components/SettingsSheet.svelte` | `'../i18n/en'` | `'../i18n'` |

In every file, change `import { t } from '...'` to point at the new path (same local name `t`).

Then change **every usage** of `t.foo` to `$t.foo` — including:
- Template text interpolations: `{t.settings}` → `{$t.settings}`
- Attribute bindings: `aria-label={t.settings}` → `aria-label={$t.settings}`
- Function-valued keys in templates: `{t.limitedData($fetchState.modelCount)}` → `{$t.limitedData($fetchState.modelCount)}` (in `App.svelte` — `$t.limitedData` is still a function on the resolved object)
- Script-side reactive statements: `$: unitLabel = t.units[unit]` → `$: unitLabel = $t.units[unit]` (in `DesktopUtilityRail.svelte` — `$t` in a `$:` statement auto-subscribes correctly in Svelte)

**`App.svelte` specific:** Has four plain-object `t.foo` usages in its template (`t.locationRequired`, `t.locationHelp`, `t.fetchingForecast`, `t.retry`) plus the function call `t.limitedData(...)` in a prop. All five must become `$t.foo`.

**`DesktopUtilityRail.svelte` specific:** The `const legendItems` array contains hardcoded English strings (`'OK'`, `'±20%'`, `'No-fly'`). These are **intentionally left untranslated** (out of scope). Do not make `legendItems` reactive.

`HeatmapCanvas.svelte`, `TimeSlider.svelte`, `ThresholdFooter.svelte`, and `ErrorBanner.svelte` do not import from `i18n/en` and require no changes.

## SettingsSheet — new Language row

`SettingsSheet.svelte` imports `{ Settings, WindUnit, AppAppearance }` from `../types`. Add `AppLanguage` to that import.

Add `const languages: AppLanguage[] = ['auto', 'en', 'pl']` in the `<script>` block, alongside the existing `const units` and `const appearances` declarations.

Insert the new section between Wind Speed Unit and Appearance:

```svelte
<div class="section">
  <div class="section-label">{$t.language}</div>
  <div class="seg-group">
    {#each languages as lang}
      <button class:active={settings.language === lang}
              on:click={() => onChange({ language: lang })}>
        {$t.languageNames[lang]}
      </button>
    {/each}
  </div>
</div>
```

`onChange({ language: lang })` follows the identical pattern used for `unit` and `appearance`.

## Data flow

```
settingsStore.language ('auto' | 'en' | 'pl')
       ↓
  resolve(lang) → Translations object
       ↓
  t  (Readable<Translations>)
       ↓
  $t.foo  ← all template expressions and attribute bindings in 5 components
```

---

## Error handling

- `resolve` guards with `typeof navigator !== 'undefined'`; falls back to `'en'`.
- An unrecognised language code (e.g. from a future locale that was removed) returns `en` via the `effective === 'pl' ? pl : en` fallback.
- Users with an older persisted settings object (no `language` key) receive `'auto'` via `defaults()` spread.

---

## Out of scope

- Translating hardcoded strings in `DesktopUtilityRail` ("Settings & Units", "Legend")
- Pluralisation rules, RTL, number/date formatting per locale
- More than two languages

---

## Testing

No new unit tests required. Existing 47 tests must continue to pass after the change.
